# System.IO.Path 设计分析：服务边界、方法正交性与信息密度

## 1. 总览

`System.IO.Path` 是一个典型的 Core Library 级别工具类。

它表面上提供了大量路径相关方法，例如：

```csharp
Path.Exists(path);
Path.Combine(path1, path2);
Path.Join(path1, path2);
Path.TryJoin(path1, path2, destination, out charsWritten);
Path.GetFileName(path);
Path.GetExtension(path);
Path.GetFullPath(path);
```

但从设计上看，它并不是简单的“路径字符串工具集合”，而是一个围绕路径语义建立的分层服务。

它的核心特点可以概括为：

```text
1. 对外 API 丰富，但内部原语克制。
2. 公开方法负责稳定语义，内部方法负责组合规则。
3. 底层方法只读取事实，不掺杂业务语义。
4. 返回值信息密度高，只返回上层刚好需要的信息。
5. 普通入口与高性能入口并存。
6. 大量使用 Span / stackalloc / ValueStringBuilder 控制分配。
```

一句话总结：

> 它不是靠“少方法”获得简洁，而是靠“少原语 + 明确分层 + 高信息密度”获得可维护性。

---

## 2. 整体分层

`Path` 的设计可以分成五层：

```text
Public API 层
  Exists / Combine / Join / TryJoin / GetFileName / GetExtension / GetFullPath

语义包装层
  null 处理、empty 处理、异常语义、root 语义、末尾 / 语义

Internal 组合层
  CombineInternal / JoinInternal / GetDirectoryNameOffset / GetFullPathInternal

底层事实层
  PathInternal.IsDirectorySeparator
  PathInternal.GetRootLength
  PathInternal.RemoveRelativeSegments

平台交互层
  Interop.Sys.LStat
  Interop.Sys.GetCwd
  Interop.Sys.MksTemps
```

这个分层的核心是：

```text
公开 API：稳定、好用、符合用户直觉
内部组合：处理路径规则
底层事实：只做最小判断
平台交互：对接操作系统
```

这类设计很适合 Core 层代码。

对于自己的项目，例如 `Pasture / FSE / BitOps / MathKit`，也可以采用类似结构：

```text
Public API:
  面向业务，语义稳定

Core Primitive:
  提供最小事实，方便组合

Low-Level Ops:
  位运算、几何、数组访问、平台细节
```

## 3. 服务边界：Public API 负责稳定语义

以 `Exists` 为例：

```csharp
public static bool Exists(string? path)
```

它对外只回答一个问题：

```text
这个路径是否存在？
```

但是内部实际处理了很多语义：

```text
path 为 null -> false
path 为空 -> false
path 非法 -> false
无法访问 -> false
路径以 / 结尾 -> 必须是目录才算存在
```

这说明公开 API 并不只是简单转发到底层系统调用，而是提供稳定、符合用户预期的服务语义。

例如：

```csharp
Path.Exists("/tmp/a.txt/")
```

如果 `/tmp/a.txt` 是文件而不是目录，那么虽然文件存在，但这个带 `/` 的路径语义上要求它是目录，所以结果应该是 `false`。

这就是 Public API 层的职责：

```text
不是只返回系统事实，而是返回 API 语义下的结果。
```

---

## 4. `ExistsCore`：高信息密度的内部事实方法

`ExistsCore` 的设计很典型：

```csharp
private static bool ExistsCore(string fullPath, out bool isDirectory)
{
    Interop.Sys.FileStatus output;
    bool flag = Interop.Sys.LStat(fullPath, out output) == 0;
    isDirectory = flag && (output.Mode & 0xF000) == 0x4000;
    return flag;
}
```

它做了两件事：

```text
1. 返回路径是否存在。
2. 通过 out 参数返回是否是目录。
```

注意，它没有返回完整的 `FileStatus`。

虽然底层已经拿到了：

```csharp
Interop.Sys.FileStatus output;
```

但它没有把整个底层结构暴露给上层，而是只提取出上层真正需要的信息：

```text
exists
isDirectory
```

这就是非常高的信息密度：

```text
底层读到复杂事实，
中间层提取最小信息，
上层用这些信息组合稳定语义。
```

`ExistsCore` 如果只返回 `bool`，信息不够，因为上层还需要判断末尾 `/` 是否对应目录。

但如果返回完整 `FileStatus`，信息又太多，会泄露底层细节。

所以它选择：

```csharp
bool ExistsCore(string fullPath, out bool isDirectory)
```

刚好够用。

## 5. 方法正交性：事实层与语义层分离

`Exists` 和 `ExistsCore` 的关系非常清楚：

```text
Exists:
  负责路径语义、异常语义、末尾分隔符语义

ExistsCore:
  负责系统事实读取：存在吗？是不是目录？
```

这就是强正交性。

它们没有互相抢职责：

```text
Exists 不直接处理 Mode 位运算。
ExistsCore 不处理 null、empty、异常策略。
```

这种分离的好处是：

```text
1. 上层语义清晰。
2. 底层事实方法可复用。
3. 修改平台实现不会影响外部 API。
4. 修改 API 语义也不需要改底层系统调用封装。
```

对应到自己的代码里，可以这样设计：

```csharp
public bool IsSelectable(int tileIndex)
{
    // 业务语义层
}

private bool TryGetTileStateCore(
    int tileIndex,
    out bool present,
    out bool visible,
    out bool selectable)
{
    // 最小事实层
}
```

不要让一个方法同时做：

```text
检查参数
读 bitset
查 tile
判断可见性
判断可选性
生成候选
计算评分
```

这会导致方法膨胀，正交性下降。

## 6. `Combine / Join / TryJoin`：同一能力的不同成本模型

`Path` 对路径拼接提供了三组能力：

```csharp
Path.Combine(...);
Path.Join(...);
Path.TryJoin(...);
```

它们看起来都在“拼路径”，但语义和成本模型不同。

---

### 6.1 Combine：路径组合语义

`Combine` 的语义更强。

如果后面的 path 是 rooted path，会覆盖前面的 path。

例如：

```csharp
Path.Combine("/a/b", "/c")
```

语义上接近：

```text
/c
```

因为第二段已经是绝对路径。

所以 `Combine` 不是单纯字符串连接，而是路径组合。

它处理的是：

```text
segment 为空
segment 为 rooted path
是否需要目录分隔符
```

---

### 6.2 Join：路径连接语义

`Join` 更接近字符串拼接：

```csharp
Path.Join(path1, path2)
```

它主要负责：

```text
把多个 path segment 连起来
必要时补 /
避免重复 /
```

它不强调 rooted path 覆盖语义。

所以：

```text
Combine 更偏路径语义
Join 更偏连接语义
```

---

### 6.3 TryJoin：零分配 / 高性能语义

`TryJoin` 的签名是：

```csharp
public static bool TryJoin(
    ReadOnlySpan<char> path1,
    ReadOnlySpan<char> path2,
    Span<char> destination,
    out int charsWritten)
```

它不返回 string，而是写入调用者提供的 buffer。

它的语义是：

```text
如果 destination 足够大，就写入并返回 true。
如果 destination 不够大，就返回 false。
```

这是一种典型的高性能 API 风格：

```text
调用者控制内存
方法只负责计算和写入
不主动分配
```

这三者可以总结为：

| 方法        | 语义                     | 分配策略      | 适用场景       |
| --------- | ---------------------- | --------- | ---------- |
| `Combine` | 路径组合，rooted path 可覆盖前面 | 返回 string | 普通业务       |
| `Join`    | 路径片段连接                 | 返回 string | 普通拼接       |
| `TryJoin` | 写入已有 buffer            | 调用方控制分配   | 热路径 / 零 GC |

这非常适合借鉴到自己的项目中。

例如 FSE 可以提供：

```csharp
// 好用版本
Behaviour[] Collect(LevelCore level);

// 高性能版本
int Collect(LevelCore level, Span<Behaviour> destination);
```

或者：

```csharp
// 好用版本
Move[] ToMoves();

// 高性能版本
int WriteMoves(Span<Move> destination);
```

## 7. Span 版本是核心逻辑，string 版本是易用包装

`Path` 很多方法都有两套重载：

```csharp
GetExtension(string? path)
GetExtension(ReadOnlySpan<char> path)

GetFileName(string? path)
GetFileName(ReadOnlySpan<char> path)

Join(string? path1, string? path2)
Join(ReadOnlySpan<char> path1, ReadOnlySpan<char> path2)
```

很多 `string` 版本内部会转成 `ReadOnlySpan<char>`：

```csharp
Path.GetExtension(path.AsSpan())
Path.GetFileName(path.AsSpan())
```

这说明真正核心逻辑通常写在 Span 版本里。

设计模式是：

```text
string 入口：
  易用、符合普通用户习惯

Span 入口：
  高性能、少分配、适合内部复用
```

这对自己的 MathKit / BitSet / FSE 都有启发。

例如：

```csharp
public static double[] Softmax(double[] scores, double temperature = 1.0)
{
    var result = new double[scores.Length];
    Softmax(scores, result, temperature);
    return result;
}

public static void Softmax(
    ReadOnlySpan<double> scores,
    Span<double> weights,
    double temperature = 1.0)
{
    // 真正核心逻辑
}
```

也就是：

```text
普通入口负责好用
Span 入口负责核心
```

## 8. 先算长度，再一次性写入

在 `Combine(params string[] paths)` 和 `Join(params string?[] paths)` 里，可以看到典型的两阶段写法：

```text
第一阶段：
  计算容量
  判断起点
  判断是否需要 separator

第二阶段：
  EnsureCapacity
  Append
```

例如 `Combine(params string[] paths)` 会先计算：

```text
最终容量 capacity
从哪个 rooted path 开始拼接
哪些片段需要追加 /
```

然后才使用：

```csharp
ValueStringBuilder valueStringBuilder = new ValueStringBuilder(stackalloc char[260]);
valueStringBuilder.EnsureCapacity(capacity);
```

这种设计的重点是：

```text
不要边写边随机扩容
尽量先知道规模
然后一次性写入
```

对热路径代码来说，这是非常重要的性能习惯。

对应到自己的项目：

```text
先 Count
再 EnsureCapacity / Rent / stackalloc
最后 Fill
```

例如候选组生成：

```csharp
int count = CountCandidates(context);
EnsureCandidateCapacity(count);
WriteCandidates(context, buffer);
```

比直接：

```csharp
List<Candidate> list = new();
foreach (...)
{
    list.Add(candidate);
}
```

更适合 Core 热路径。

## 9. 异常策略只放在语义层

`Exists` 会捕获部分异常：

```text
ArgumentException
IOException
UnauthorizedAccessException
```

然后返回 `false`。

这符合 `Exists` 的 API 语义：

```text
无法确认存在，就返回 false。
```

但 `ExistsCore` 不处理这些异常策略。

它只假设：

```text
输入已经是 fullPath
我只负责调用 lstat 并返回事实
```

这说明：

```text
异常策略属于 Public API / 语义层
底层 Core 方法不负责复杂异常语义
```

这点在自己的代码里也应该坚持。

例如：

```csharp
public bool TrySelectTile(int tileIndex)
{
    // 可以处理非法输入、状态异常、业务规则
}

private bool IsSelectableCore(int tileIndex)
{
    // 只读 bitset / 状态，不处理复杂业务异常
}
```

## 10. 命名风格：朴素但层级清楚

`Path` 的命名并不花哨：

```csharp
Exists
ExistsCore
CombineInternal
JoinInternal
GetDirectoryNameOffset
GetFullPathInternal
```

这些名字的特点是：

```text
1. Public 方法直接表达用户意图。
2. Internal 方法表达其作为内部实现的身份。
3. Core 方法表达它是某个语义的底层事实版本。
```

`ExistsCore` 这个名字尤其典型：

```csharp
bool ExistsCore(string fullPath, out bool isDirectory)
```

它说明：

```text
主语义是 Exists
但它是内部核心
同时附带一个上层需要的最小事实 isDirectory
```

这比下面这些名字更克制：

```csharp
TryGetFileSystemStatus
CheckPathExistenceAndDirectoryType
GetPathInfo
```

因为它没有扩大概念边界。

自己的项目里也可以采用类似命名：

```csharp
IsSelectable(...)
IsSelectableCore(...)

CollectCandidates(...)
CollectCandidatesCore(...)

TryGetFirstHit(...)
TryGetFirstHitCore(...)
```

但要注意：

```text
Core / Internal 不应该滥用。
只有当它确实承担“底层核心实现”时才使用。
```

## 11. 信息密度：不是返回越多越好

一个常见错误是底层方法返回一个很大的信息对象：

```csharp
public readonly struct TileStateInfo
{
    public bool Present;
    public bool Visible;
    public bool Selectable;
    public bool Blocked;
    public int Color;
    public int Z;
    public int Region;
    public int BlockerCount;
}
```

这种对象看似“信息完整”，但会带来几个问题：

```text
1. 上层不知道哪些字段可靠。
2. 字段之间可能有隐含依赖。
3. 很容易让底层方法承担过多职责。
4. 后续维护时容易破坏不变量。
```

`ExistsCore` 的方式更好：

```csharp
bool exists = ExistsCore(fullPath, out bool isDirectory);
```

它只返回上层马上需要的事实。

对应到自己的 `Pasture`：

```csharp
bool TryGetFirstVerticalHit(
    int tileIndex,
    NeighborDir dir,
    int basePosition,
    out int hitTileIndex)
```

这比返回大对象更克制：

```csharp
VerticalHitInfo
{
    bool HasHit;
    int TileIndex;
    int Color;
    int Z;
    bool IsSelectable;
    bool IsVisible;
}
```

除非上层真的需要这些信息，否则不要提前扩大返回值。

## 12. 方法数量多不等于设计混乱

`Path` 的方法很多，但它们可以被清晰分组：

```text
路径切片：
  GetDirectoryName
  GetFileName
  GetExtension
  GetFileNameWithoutExtension

路径拼接：
  Combine
  Join
  TryJoin

路径规范化：
  GetFullPath
  GetRelativePath
  TrimEndingDirectorySeparator

路径判断：
  Exists
  HasExtension
  IsPathRooted
  IsPathFullyQualified
  EndsInDirectorySeparator

临时路径：
  GetTempPath
  GetTempFileName
  GetRandomFileName
```

所以关键不是“方法少”，而是：

```text
方法族是否清晰
同一族方法是否语义一致
底层原语是否共享
```

如果每个方法都有自己的规则，那就会乱。

如果多个方法共享少量原语，那方法多也可以很稳。

## 13. 对 ThreeTile Core 的设计启发

结合你的项目，最值得借鉴的是下面几条。

---

### 13.1 Public API 语义化

业务层应该看到类似这样的 API：

```csharp
pasture.IsPresent(tileIndex);
pasture.IsVisible(tileIndex);
pasture.IsSelectable(tileIndex);
pasture.TryGetFirstBlocker(tileIndex, dir, out int blockerIndex);
pasture.CountSelectableSameColor(color);
```

而不是到处写：

```csharp
BitOps.Contains(_present, tileIndex);
BitOps.Contains(_selectable, tileIndex);
_geo.ProjectColumn(...);
```

底层 bitset 和 geometry 可以存在，但不应该随意泄露给业务层。

---

### 13.2 Core Primitive 返回最小事实

类似：

```csharp
private static bool ExistsCore(string fullPath, out bool isDirectory)
```

可以设计：

```csharp
private bool TryGetTileStateCore(
    int tileIndex,
    out bool present,
    out bool selectable)
```

或者：

```csharp
private bool TryGetFirstHitCore(
    int tileIndex,
    NeighborDir dir,
    int columnBase,
    out int hitTileIndex)
```

只返回上层组合语义需要的信息。

---

### 13.3 同一能力提供不同成本模型

可以参考 `Join / TryJoin`。

例如候选收集：

```csharp
public Behaviour[] Collect(LevelCore level);
public int Collect(LevelCore level, Span<Behaviour> destination);
```

路径展开：

```csharp
public Move[] ToMoves(Behaviour behaviour);
public int WriteMoves(Behaviour behaviour, Span<Move> destination);
```

评分：

```csharp
public double[] Softmax(double[] scores);
public static void Softmax(ReadOnlySpan<double> scores, Span<double> weights);
```

这能同时满足：

```text
1. 普通调用好写。
2. 热路径可控分配。
```

---

### 13.4 先 Count，再 Fill

对于 FSE / Finder / Metrics，尽量形成：

```csharp
int count = CountXxx(context);
EnsureCapacity(count);
FillXxx(context, destination);
```

或者：

```csharp
int written = WriteXxx(context, destination);
```

这和 `Path.Combine` 的思想一致：

```text
先算规模
再准备空间
最后一次性写入
```

---

### 13.5 避免业务层直接依赖底层结构

不要让业务层到处知道：

```text
ulong[] present
ulong[] selectable
wordCount
color * wordCount + wordIndex
region offset
packed xyz
```

这些应该被封装在更低层。

业务层最好面对：

```text
tileIndex
color
dir
count
hit
selectable
```

这可以明显降低心智负担。

## 14. 可套用的 Core API 设计模板

可以形成下面这种固定模板：

```csharp
public sealed class Xxx
{
    // 1. Public 语义 API
    public bool HasSomething(int id)
    {
        if (!IsValid(id))
            return false;

        return HasSomethingCore(id);
    }

    // 2. Internal / Core 原始事实方法
    private bool HasSomethingCore(int id)
    {
        return BitOps.Contains(_bits, id);
    }

    // 3. Try 写入型高性能 API
    public bool TryWriteSomething(
        int id,
        Span<int> destination,
        out int written)
    {
        written = 0;

        int count = CountSomethingCore(id);
        if (destination.Length < count)
            return false;

        written = FillSomethingCore(id, destination);
        return true;
    }

    // 4. Count / Fill 分离
    private int CountSomethingCore(int id)
    {
        // 只计算数量
    }

    private int FillSomethingCore(int id, Span<int> destination)
    {
        // 只负责写入
    }
}
```

这种结构的优点是：

```text
Public API 好用
Core 方法正交
热路径可控
返回信息克制
```

## 15. 最终总结

`System.IO.Path` 的整体设计价值，不在于某个单独方法写得多漂亮，而在于它的整体结构很成熟。

它体现的是：

```text
1. 服务边界清楚：
   Public API 负责稳定语义。

2. 方法正交性强：
   事实读取、路径组合、异常语义、平台交互分层处理。

3. 信息密度高：
   内部方法只返回上层需要的最小事实。

4. 成本模型丰富：
   string 版本好用，Span / Try 版本高性能。

5. 分配控制明确：
   先计算容量，再一次性写入。

6. 底层复杂性不泄露：
   Interop / FileStatus / Mode 位运算都被封装在 Core 内部。
```

对自己的项目来说，最重要的一句话是：

> Core 层不是方法越少越好，而是原语要少、职责要正交、信息要刚好够用、业务语义不要泄露到底层。
